# A2A package assessment

### 1. What the package actually is

This package is a local-first terminal collaboration bridge for coding agents. Its core assumption is not that agents expose interoperable service APIs. Its assumption is that the useful agents already live inside interactive terminal CLIs, usually Claude Code, Codex, Gemini, or Cursor Agent, and that the highest-leverage way to let them collaborate is to make those terminal sessions addressable.

The load-bearing runtime is a small HTTP server in [src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:14), where the bridge keeps an in-memory `Map` of registered agents. A registration is an `agentId`, a tmux target such as `bob:0.0`, optional cwd and description metadata, backend launch metadata, and optional `bridgeUrl` for remote forwarding. The actual endpoints are all in [src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:180): `POST /api/a2a/register`, `DELETE /api/a2a/register/:id`, `GET /api/a2a/agents`, `POST /api/a2a/send`, plus `/health`.

The message model is deliberately thin. A send request must contain `to`, `from`, `origin`, and string `body`; `origin` is limited to `user`, `peer`, or `self`; `action` is semantic metadata defaulting to `message` and normally one of `message`, `reply`, or `ask` ([src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:215), lines 215-243). Local delivery wraps the body in an XML-like `<a2a_message>` envelope with attributes for sender, recipient, origin, action, extras, and timestamp, then puts the body inside CDATA ([src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:47), lines 47-56). That envelope is pasted into the recipient's tmux pane and submitted with Enter ([src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:101), lines 101-135). This is the product's deepest design choice: delivery is not a model API call, a tool call, or a protocol-level task submission. It is terminal input injection into a live agent session.

The bridge has real operational hardening around that fragile boundary. It limits request bodies to 1 MB ([src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:16), lines 16-28), serializes delivery per tmux target to prevent paste and Enter sequences from interleaving ([src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:63), lines 63-80), waits for paste settlement, and retries Enter while checking whether the TUI still shows a pasted-text placeholder ([src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:82), lines 82-135). Those details are not incidental. They are the difference between a cute demo and a usable tmux bridge.

The CLI in [src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:28) is the product shell and orchestration layer. It starts and stops the bridge, spawns agents in tmux, registers existing sessions, sends messages, lists and reconnects peers, peeks and attaches to panes, follows the message log, manages simple bearer-token auth, starts ngrok for remote sessions, starts groups from persona files, and starts teams from YAML or JSON specs. The messaging path is split across three parser styles: legacy subcommands and `to:` fields handled inside `cli.mjs`, flag-form sends handled by [src/a2a-argv.mjs](/Users/op/Documents/dev/a2a/src/a2a-argv.mjs:38), and colon-form sends handled by [src/a2a-tokens.mjs](/Users/op/Documents/dev/a2a/src/a2a-tokens.mjs:53). The current main dispatch order is colon flags first, then flag sends, then legacy forms, then command subcommands ([src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:1434), lines 1434-1488).

The config and lightweight persistence layer live in [src/a2a-config.mjs](/Users/op/Documents/dev/a2a/src/a2a-config.mjs:6). User config is stored under `~/.claude/skills/a2a/config.json`; the throwaway parser registry cache and message log live in the same skill directory; groups and teams are read from `~/.claude/skills/a2a/groups` and `~/.claude/skills/a2a/teams` ([src/a2a-config.mjs](/Users/op/Documents/dev/a2a/src/a2a-config.mjs:6), lines 6-24). Runtime agent registry state is not persisted by the bridge. `a2a reconnect` reconstructs it from live tmux sessions and the cached registry list ([src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:1204), lines 1204-1291).

Groups are simple directories of Markdown role prompts, discovered by filename and started as multiple tmux sessions ([src/a2a-config.mjs](/Users/op/Documents/dev/a2a/src/a2a-config.mjs:123), lines 123-145; [src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:895), lines 895-946). Teams are richer declarative specs loaded from JSON or a hand-rolled YAML subset in [src/a2a-team-spec.mjs](/Users/op/Documents/dev/a2a/src/a2a-team-spec.mjs:27), then normalized into per-agent backend, model, approval, sandbox, cwd, env, role, and raw args ([src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:532), lines 532-579). The common team settings are translated per backend instead of blindly passing Claude flags to every CLI ([src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:353), lines 353-431).

The optional MCP sidecar in [src/a2a-channel.mjs](/Users/op/Documents/dev/a2a/src/a2a-channel.mjs:24) is a separate product surface. It runs an MCP stdio server, exposes a `reply` tool that shells out to `a2a`, mirrors events over SSE, accepts HTTP POST bodies into Claude's `notifications/claude/channel`, and relays permission decisions from HTTP bodies like `yes abcde` or `no abcde` ([src/a2a-channel.mjs](/Users/op/Documents/dev/a2a/src/a2a-channel.mjs:67), lines 67-158; [src/a2a-channel.mjs](/Users/op/Documents/dev/a2a/src/a2a-channel.mjs:162), lines 162-235). This channel is not the bridge; it is a Claude-specific sidecar that lets external processes push text into a live Claude Code session and lets that session send bridge messages through a tool.

Installation is handled by [scripts/install.mjs](/Users/op/Documents/dev/a2a/scripts/install.mjs:361). It creates symlinks for `a2a` and `a2a-server`, checks that the skill and welcome doc exist in the package, copies bundled sample groups and teams into package-local directories, writes the session-start hook under the package's `hooks/`, updates `~/.claude/settings.json` to call that hook, and appends an a2a instruction block to package-root `CLAUDE.md` ([scripts/install.mjs](/Users/op/Documents/dev/a2a/scripts/install.mjs:361), lines 361-421; [scripts/install.mjs](/Users/op/Documents/dev/a2a/scripts/install.mjs:424), lines 424-581; [scripts/install.mjs](/Users/op/Documents/dev/a2a/scripts/install.mjs:694), lines 694-794). The docs currently describe parts of this differently, which matters later.

### 2. What A2A v1.0 actually is

Agent2Agent v1.0 is an interoperability protocol for independent, potentially opaque agent systems. The spec's opening goal is explicit: agents should discover capabilities, negotiate modalities, manage collaborative tasks, and exchange information without needing access to each other's internal state, memory, or tools ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 489-495). Its guiding principles include reuse of HTTP, JSON-RPC, and SSE, enterprise authentication and authorization, async-first long-running tasks, modality agnosticism, and opaque execution ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 503-508).

The protocol is layered. The canonical data model defines objects such as Task, Message, AgentCard, Part, Artifact, and Extension; abstract operations define actions such as Send Message, Stream Message, Get Task, List Tasks, Cancel Task, and Get Agent Card; protocol bindings map those semantics to JSON-RPC, gRPC, HTTP+JSON, or custom bindings ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 512-562). The normative source is the protobuf definition, and derived schemas or SDKs are supposed to be regenerated rather than hand-edited ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 563-565).

The core unit is the `Task`. A task has an ID, optional `contextId`, status, artifacts, history, and metadata ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 1172-1181). Task status has lifecycle states such as submitted, working, completed, failed, canceled, input required, rejected, and auth required ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 1190-1202). Messages are communication units associated with optional context and task IDs; they have a creator-generated `messageId`, role, parts, metadata, extensions, and reference task IDs ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 1203-1214). Roles are deliberately client/server shaped: `ROLE_USER` and `ROLE_AGENT` ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 1215-1222).

A2A separates communication from results. Messages initiate tasks, clarify inputs, give status, and continue task interactions; results should be returned as artifacts associated with a task, and messages must not be treated as a reliable delivery mechanism for critical information ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 1153-1166). This is a major difference from the package, whose message body is the whole product-level payload.

The operation set is broad. `SendMessage` starts an interaction and returns either a Task or direct Message ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 607-626). `SendStreamingMessage` establishes a real-time stream where either one direct message is returned or a task lifecycle stream emits status and artifact updates until terminal state ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 627-652). `GetTask`, `ListTasks`, `CancelTask`, and `SubscribeToTask` provide retrieval, pagination, lifecycle control, and update subscription ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 653-686 and 713-761). Push notification config operations let agents send webhook updates for long-running tasks ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 762-779).

The update model is three-track: polling with Get Task, streaming via Stream Message or Subscribe to Task, and push notifications via HTTP webhooks ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 1069-1097). Streaming event ordering is specified, including multiple concurrent subscribers per task receiving the same events in the same order ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 1098-1117).

Discovery is via Agent Cards. An Agent Card is a self-describing manifest for identity, supported interfaces, provider, version, documentation, capabilities, security schemes, default input and output media types, skills, signatures, and icon URL ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 1319-1337). Capabilities include streaming, push notifications, extensions, and extended Agent Card support ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 1345-1352). Skills are descriptive units with IDs, names, descriptions, tags, examples, input and output modes, and optional security requirements ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 1362-1373). Interfaces bind URL, protocol binding, optional tenant, and protocol version ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 1374-1381). Cards may be signed with JWS, with canonicalization and verification rules ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 2316-2423), and discovered at `.well-known/agent-card.json` ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 3521-3537).

The security surface is enterprise-web shaped. Security schemes mirror OpenAPI security schemes: API key, HTTP auth, OAuth2, OpenID Connect, and mTLS ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 1391-1447). Production deployments must use encrypted transport, and servers must authenticate every incoming request according to the declared requirements ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 2206-2229). Authorization is implementation-specific but must scope task access and avoid leaking resources outside caller boundaries ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 2230-2237 and 3310-3333). The general security section calls for input validation, request and file size limits, credential secrecy and rotation, auth failure logging, rate limiting, audit trails, and privacy-aware retention ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 3392-3428).

JSON-RPC is one official binding. It uses JSON-RPC 2.0 over HTTP(S), `application/json`, PascalCase method names, and SSE for streaming ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 2511-2520). It transmits A2A service parameters like `A2A-Version` and `A2A-Extensions` as HTTP headers ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 2521-2537). The spec also defines IANA-oriented media type and header registrations for `application/a2a+json`, `A2A-Version`, `A2A-Extensions`, and the well-known Agent Card URI ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 3436-3504 and 3505-3540).

The migration appendix is not minor housekeeping. It shows v1.0 deliberately moved away from inline `kind` discriminators toward member-name discriminators for Part and streaming event unions, and moved extended Agent Card support into capabilities ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 3543-3623 and 3653-3708). That reinforces the spec's model: typed data structures and explicit compatibility management, not raw conversational envelopes.

### 3. Structural comparison

| Axis | This package | A2A v1.0 | Assessment |
|---|---|---|---|
| Communication shape | Natural-language text envelope pasted into a live tmux session. The peer agent sees it as a user turn. | Structured protocol operations over bindings. Messages initiate or continue tasks; outputs are artifacts. | The overlap is intent, not architecture. Both enable agent collaboration, but one drives terminal CLIs and the other defines service interoperability. |
| Primary object | Registered live session: `agentId`, `tmuxTarget`, optional `bridgeUrl`. | Task, Message, Part, Artifact, AgentCard, SecurityScheme. | The package models "who can I paste into"; A2A models "what task is being performed and how clients can manage it." |
| Persistence | Bridge registry in memory; cached agent/group names in `registry.json`; human-readable message log. | Task state, history, artifacts, push configs, and Agent Cards are protocol-visible concepts. | The package is intentionally ephemeral, with reconnect as repair. A2A assumes task state is retrievable and manageable. |
| Transport | Local HTTP to bridge, tmux paste-buffer for local delivery, HTTP forwarding to remote bridge URL, optional ngrok. | JSON-RPC, gRPC, HTTP+JSON, and custom bindings, generally over HTTPS/TLS. | The package's actual transport boundary is the terminal. A2A's boundary is a network API. |
| Trust and identity | Optional shared bearer key and peer keys in config. If no key and no peers, bridge is open. `from` is caller-supplied. | Declared security schemes, authenticated requests, authorization scoping, TLS, signatures for Agent Cards. | The package has enough auth for trusted local use and ad hoc remote use. It does not have protocol-grade identity or authorization. |
| Discovery | `GET /api/a2a/agents` lists live registry entries. Groups/teams are local files. | `.well-known/agent-card.json`, Agent Cards, capabilities, skills, interfaces, versions, security schemes, caching, signatures. | The package discovers sessions; A2A discovers services and their interaction contract. |
| Multi-modal content | Raw string body only. Extra attributes can ride on the envelope, but content is text. | Parts include structured content and file/data forms; input and output modes are media types; artifacts carry outputs. | The package is correct to stay text-first for terminal agents, but it cannot interoperate with typed multi-modal clients. |
| Capability negotiation | None beyond human-readable group/team roles and backend selection. | Capabilities and skills declared on Agent Cards; extensions and protocol versions negotiated. | This is one of the biggest non-overlaps. The package relies on humans/agents reading prompts; A2A relies on machine-readable contracts. |
| Error model | `{ success, data?, error?, timestamp? }` plus CLI exits and stderr. | Binding-specific error mappings with standard and A2A-specific errors and structured error details. | The package is pragmatic, but error semantics are not portable or typed. |
| Versioning | Package version in `package.json`; no wire-format version in bridge messages. | `A2A-Version` service parameter and versioned protocol interfaces. | Fine for a local tool; weak for remote compatibility once third parties depend on the bridge. |
| Execution boundary | Agents are not opaque services; they are visible panes, and `peek`/`attach` are first-class. | Opaque execution is a guiding principle; clients should not need agent internals. | This package deliberately violates A2A's opacity for practical collaboration. That is a product strength, not a spec failure. |
| Human-in-loop | Native, because everything is an interactive terminal session. | Native through task states, auth-required/input-required states, streaming, and push notifications. | Both care about long-running work, but the package makes humans operators of panes; A2A makes clients managers of tasks. |

The closest conceptual overlap is "send a message to another agent." In this package, a send is an HTTP bridge call whose successful completion means the target terminal received an envelope and Enter was sent. In A2A, `SendMessage` means the receiving agent accepted a typed request and returned either a direct message or a task handle. That difference cascades into everything else. The package cannot naturally answer "what is the task status?" because it has no task object. A2A cannot naturally answer "what is Bob's pane showing right now?" because that is deliberately outside its abstraction.

### 4. What's good in the code

The strongest thing in the package is that it has found the right integration level for today's coding agents. Instead of pretending every agent CLI is already a clean API server, it treats tmux as the substrate and makes live sessions routable. The delivery path in [src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:101), lines 101-135, is ugly in the way real systems are ugly: load a tmux buffer, paste it, wait for the TUI to settle, send Enter, and verify the placeholder disappeared. That is exactly the kind of small operational knowledge a competitor would underestimate.

The per-target delivery lock is also a real strength. The comment explains the failure mode precisely: once delivery became async, concurrent sends to the same pane could interleave paste-buffer and Enter sequences ([src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:63), lines 63-80). This is not over-engineering. It is a minimal concurrency guard at the one place where ordering actually matters.

The envelope hardening is better than the old bug catalog indicates. The body is now CDATA-wrapped and splits embedded `]]>` safely ([src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:49), lines 49-56). Request bodies are capped at 1 MB ([src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:16), lines 16-28). These are small changes, but they show the product is being hardened at the correct boundary.

The CLI's "session dead, attempt restart" behavior is product-smart. Before sending to a registered local target, both `sendNormalizedEnvelope` and flag-form send check whether the tmux session still exists and try to restart using stored backend args and env ([src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:644), lines 644-690; [src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:715), lines 715-761). That preserves the user's mental model: registered peers are people you can reach, not brittle process IDs.

The group and team layers are strategically good. Groups let the product exploit prompt/persona files as swarm material with almost no machinery ([src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:895), lines 895-946). Team specs add just enough structure to express mixed backends, approval modes, sandboxes, cwd, env, raw args, and role prompts ([src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:532), lines 532-579). The `bug-killers` sample shows a real debugging workflow with scout, reproducer, patcher, and verifier roles ([teams/bug-killers.yaml](/Users/op/Documents/dev/a2a/teams/bug-killers.yaml:1), lines 1-36). The `west-world` sample shows the system is not only a coding swarm launcher; it can run a large creative room with role specialization and a persistent artifact ([teams/west-world/west-world.yaml](/Users/op/Documents/dev/a2a/teams/west-world/west-world.yaml:1), lines 1-39; [teams/west-world/westworld-season-5-ending.md](/Users/op/Documents/dev/a2a/teams/west-world/westworld-season-5-ending.md:1), lines 1-15).

The backend abstraction is modest and useful. `translateCommonAgentSettings` maps common concepts like model, approval, and sandbox into the different flag dialects of Claude, Codex, Gemini, and Cursor Agent ([src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:353), lines 353-431). That solves a real user problem without trying to create a universal agent runtime.

The installer is cautious in places that matter. Binary installation uses symlinks and refuses to replace conflicting files ([scripts/install.mjs](/Users/op/Documents/dev/a2a/scripts/install.mjs:317), lines 317-339). Settings updates back up existing JSON and merge a nested SessionStart hook instead of overwriting the whole file ([scripts/install.mjs](/Users/op/Documents/dev/a2a/scripts/install.mjs:694), lines 694-765). This fits a tool that lives in a user's development environment, where reckless setup scripts are expensive.

The a2a skill and welcome doc correctly teach the behavioral trap. The important user experience is not "run a command"; it is "a peer will not see your normal chat response unless you send through the bridge." The skill states that directly ([skill/SKILL.md](/Users/op/Documents/dev/a2a/skill/SKILL.md:112), lines 112-116) and pushes agents away from empty acknowledgements toward substantive results ([skill/SKILL.md](/Users/op/Documents/dev/a2a/skill/SKILL.md:12), lines 12-24). That is product design, not just documentation.

### 5. What's wrong, in priority order

The highest-priority issue is not that the package fails to implement A2A v1.0. It does not try to. The highest-priority issue is that the name and public framing collide with a now-formal standard whose mental model is very different. `package.json` publishes the package as `a2a` ([package.json](/Users/op/Documents/dev/a2a/package.json:1), lines 1-9), the README title is `a2a` ([README.md](/Users/op/Documents/dev/a2a/README.md:1), lines 1-5), and the CLI uses `/api/a2a/*` endpoints ([docs/cli.md](/Users/op/Documents/dev/a2a/docs/cli.md:128), lines 128-140). The package's substance is "terminal bridge for live coding-agent sessions," while A2A v1.0 is "task protocol for opaque agent services." The collision will create false expectations from standards-aware users.

The second priority is identity and trust. The bridge accepts unauthenticated requests whenever no key and no peers are configured ([src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:137), lines 137-147). That is defensible for localhost use, but `a2a start-global` exposes bridges through ngrok ([src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:1032), lines 1032-1083), and the system still has caller-supplied `from`, `origin`, and `action` fields. Auto-registration via `replyTo` is now restricted to authenticated peers whose token maps to `body.from` ([src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:229), lines 229-230), which fixes the specific old bug. But `/api/a2a/register` itself still accepts arbitrary `agentId`, `tmuxTarget`, and `bridgeUrl` from any authorized caller, and if the bridge is open, from any caller who can reach it ([src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:183), lines 183-198). At scale, the trust model needs to distinguish local operator control, remote peer self-registration, and untrusted callers.

The third priority is that the runtime registry is memory-only while several user-visible concepts behave as if registration is durable. The bridge registry is a process-local `Map` ([src/a2a-server.mjs](/Users/op/Documents/dev/a2a/src/a2a-server.mjs:14)), and the docs correctly note that reconnect repairs state after bridge restart ([docs/cli.md](/Users/op/Documents/dev/a2a/docs/cli.md:163), lines 163-165). Reconnect is a good repair tool, but the product is now doing more than two local panes. It has remote bridge URLs, teams, dashboards, and restart metadata. At that point, "registry disappears on bridge restart" becomes not just an implementation detail but a product constraint.

The fourth priority is parser and command-surface overlap. There are at least four parsing paths: generic `parseArgs` in [src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:133), auth-specific `parseAuthArgs` in [src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:154), flag-form send parsing in [src/a2a-argv.mjs](/Users/op/Documents/dev/a2a/src/a2a-argv.mjs:38), and colon-form parsing in [src/a2a-tokens.mjs](/Users/op/Documents/dev/a2a/src/a2a-tokens.mjs:53). They each have different assumptions about unknown flags, boolean flags, values, recipients, and reserved keys. This has already produced real bugs in `BUGS.md`. One remains open in flag-form parsing: any unknown bare flag becomes a recipient ([src/a2a-argv.mjs](/Users/op/Documents/dev/a2a/src/a2a-argv.mjs:78), lines 78-84). I verified the current behavior with Node: `parseFlagSendArgv(['--verbose','--sam','hello'])` returns recipients `['verbose','sam']`.

The fifth priority is file size and coupling. [src/cli.mjs](/Users/op/Documents/dev/a2a/src/cli.mjs:1) is 1,488 lines and owns unrelated concerns: HTTP client, tmux management, backend launch, persona composition, team normalization, ngrok, parser dispatch, session lifecycle, message sending, logging, config, auth, and output formatting. [scripts/install.mjs](/Users/op/Documents/dev/a2a/scripts/install.mjs:1) is 932 lines and owns install path detection, symlinks, file copying, hook source generation, settings JSON surgery, CLAUDE.md mutation, prereq reporting, and verification. These are not cosmetic size complaints. The coupling makes it harder to harden trust, replace parsers, or change installation semantics without breaking far-away behavior.

The sixth priority is the hand-rolled YAML parser. [src/a2a-team-spec.mjs](/Users/op/Documents/dev/a2a/src/a2a-team-spec.mjs:27), lines 27-199, implements comment stripping, scalar parsing, block scalars, mappings, sequences, and indentation rules manually. It exists to support team specs, which are strategic to the product. A YAML subset can be fine if it is named and tested as a subset. Right now it looks like YAML support while silently omitting much of YAML's real behavior. That will bite because team specs are exactly where users will want comments, quoted strings, arrays, nested env, role files, and multiline text.

The seventh priority is docs/install drift. README says `npm run bootstrap` copies `skill/` into `~/.claude/skills/a2a/` and installs a welcome doc ([README.md](/Users/op/Documents/dev/a2a/README.md:77), lines 77-84). `docs/cli.md` says the install script installs the skill and welcome doc under `~/.claude` ([docs/cli.md](/Users/op/Documents/dev/a2a/docs/cli.md:41), lines 41-49). Current installer code says the skill and welcome doc stay in the package and are not copied ([scripts/install.mjs](/Users/op/Documents/dev/a2a/scripts/install.mjs:424), lines 424-455). AGENTS.md also says bootstrap writes skill and welcome doc under `~/.claude` ([AGENTS.md](/Users/op/Documents/dev/a2a/AGENTS.md:25), lines 25-33). This is not cosmetic. It changes what a user can expect after setup and where other agents should look for the skill.

The eighth priority is logging and privacy. `appendMessageLog` writes every attempted body to a plaintext log by default under `~/.claude/skills/a2a/messages.log` ([src/a2a-config.mjs](/Users/op/Documents/dev/a2a/src/a2a-config.mjs:179), lines 179-215). This is useful. It is also a retention policy, and the package does not treat it as one. The A2A spec explicitly calls out sensitive data in task history and artifacts, audit trails, and retention policies ([A2A spec](https://a2a-protocol.org/latest/specification/), lines 3413-3428). For local trusted use, plaintext logs are acceptable. For remote peers and real work, the product should make log retention and redaction explicit.

The ninth priority is that `a2a-channel` has its own trust surface and no body-auth symmetry with the bridge. It gates HTTP POSTs with an `X-Sender` allowlist defaulting to `dev` ([src/a2a-channel.mjs](/Users/op/Documents/dev/a2a/src/a2a-channel.mjs:35), lines 35-40; [src/a2a-channel.mjs](/Users/op/Documents/dev/a2a/src/a2a-channel.mjs:183), lines 183-190), then forwards body text into Claude channel notifications. It also exposes SSE to anyone who can connect to the host/port ([src/a2a-channel.mjs](/Users/op/Documents/dev/a2a/src/a2a-channel.mjs:165), lines 165-180). Bound to `127.0.0.1`, that is fine. If users change `A2A_CHANNEL_HOST`, it becomes a separate ingress path into a live agent session with a very lightweight sender check.

### 6. Where it overlaps with A2A v1.0 and where it doesn't

| A2A v1.0 concept | Package concept | Match quality | Notes |
|---|---|---|---|
| Agent | Registered tmux session | Partial | A2A agent is an opaque service endpoint; package agent is a live CLI pane. |
| Agent Card | `a2a list` registry entry plus local docs/personas | Weak | Registry entries have identity and target metadata, not capabilities, skills, input/output modes, security schemes, or versions. |
| Agent Skill | Group/team role prompt | Weak but interesting | Team roles describe likely behavior for humans/agents, not machine-negotiable capabilities. |
| SendMessage | `POST /api/a2a/send` | Partial | Both send text to an agent. A2A returns a Task or Message; the package returns delivery status. |
| Message | `<a2a_message>` envelope | Partial | Package messages are raw text plus envelope attributes. A2A messages are typed objects with IDs, roles, parts, context/task association, extensions, and metadata. |
| Task | None | No match | The package has no task lifecycle, status, cancellation, history, or artifacts. |
| Context ID | tmux session / conversation pane | Weak | The pane contains conversational context, but it is not represented as a protocol ID. |
| Artifacts | Files produced by agents in workspace, or text bodies in messages | Weak | The package relies on agents to create files or paste results; it does not model artifacts. |
| Streaming | tmux live pane, `peek`, optional channel SSE | Weak | Live pane observation is operational, not task event streaming. Channel SSE mirrors sidecar text, not task updates. |
| Push notifications | `a2a-channel` HTTP POST into Claude channel | Weak | Both are webhook-like, but A2A push notifications are task update payloads with auth; channel POSTs are inbound text events. |
| Discovery | `/api/a2a/agents` | Partial | Discovers who is registered now, not what they can do or how to call them interoperably. |
| SecurityScheme | Config key and peer keys | Weak | Bearer token support exists, but there is no declared scheme object, scopes, OAuth/OIDC/mTLS, or per-operation authorization. |
| Protocol binding | HTTP bridge API plus tmux delivery | No A2A binding | A custom binding would need equivalent A2A data model and all core operations; this package intentionally does not provide that. |
| Versioning | `package.json` version | Weak | No `A2A-Version` wire header or interface version negotiation. |
| Extensions | Extra envelope attributes | Weak | Extras can be carried, but no URI-based extension declaration or negotiation. |

A2A v1.0 solves things this package does not: machine-readable agent discovery, capability and skill declaration, typed multi-modal content, durable task state, artifacts, task history, pagination, cancellation, streaming task events, push notification lifecycle, service-level auth and authorization, signed Agent Cards, version negotiation, formal error semantics, and interoperability across vendors.

This package solves things A2A v1.0 does not: addressing real interactive coding-agent CLI sessions, using tmux as a shared collaboration substrate, letting agents inspect or attach to each other's panes, spinning up named peer sessions from role prompts, creating mixed-backend swarms from local team specs, preserving the human operator's ability to watch and intervene, and making collaboration work without waiting for every agent vendor to expose an A2A server.

### 7. Strategic options

| Strategy | When it is right | Cost | What it gives up |
|---|---|---|---|
| Terminal bridge, proudly not A2A | Best when the target users are operators running local or remote coding-agent CLIs, and the product's unique value is live collaboration inside tmux. | Rename/reframe public surfaces, harden trust, simplify parser and registry internals, improve docs, add tests. | Gives up standards compliance as a near-term goal and accepts that A2A-aware users need a compatibility story later. |
| A2A adapter at the edge | Best when the project wants to participate in the A2A ecosystem while keeping tmux delivery as the core engine. | Add Agent Card endpoint, version headers, a small subset of A2A operations, mapping `SendMessage` to bridge sends, and maybe synthetic Tasks for delivery tracking. Requires careful disclaimers. | Gives up purity. The adapter would be a bridge facade, not full A2A semantics unless much more state is added. |
| Full A2A-compatible agent host | Best only if the goal changes from "coordinate coding CLIs" to "be an interoperable agent service platform." | Major rewrite-level cost: persistent task store, artifacts, Agent Cards, auth/authorization, versioning, typed parts, streaming, push configs, error model, conformance tests. | Gives up the simplicity and immediacy that make the current product useful. The tmux bridge becomes an implementation detail instead of the product. |
| Swarm launcher and operator console | Best if the team/group feature is the real product: run many specialized agents, watch them, route work, converge on an artifact. | Invest in team spec schema, dashboards, durable run state, role libraries, logging controls, and orchestration ergonomics. | Gives up being primarily a messaging primitive. It becomes an operator product with more UX responsibility. |

### 8. Concrete recommendation

Pick "Terminal bridge, proudly not A2A," with a small future opening for an A2A adapter. The product's strongest asset is not standards alignment. It is that it works today with the agents people actually use in terminals. Trying to become full A2A v1.0 now would spend the project's energy rebuilding task protocol machinery while weakening the weird, useful thing it already does well. The name collision is real, but the correct move is not to chase the spec. The correct move is to clarify the product boundary and harden the bridge.

The next moves I would do, in order:

1. Rename the public framing before deepening the API. Keep the CLI name if you want muscle memory, but make the product name and README say "a2a-bridge" or "terminal agent bridge" everywhere. Add one explicit paragraph: this is not an implementation of Agent2Agent v1.0; it is a tmux bridge for live agent CLIs.

2. Fix the remaining parser bug by making flag-form recipients registry-aware or by requiring unknown bare flags to be explicit recipients only under a documented mode. Then collapse the send parsers behind one normalized envelope shape.

3. Decide the trust model for three cases: local-only open bridge, authenticated remote peer, and operator/admin. Enforce that distinction in `/register` and `/send`. Make `start-global` warn or refuse when exposing an unauthenticated bridge.

4. Resolve docs/install drift. Either actually install the skill and welcome doc under `~/.claude`, or update README, docs, and AGENTS.md to say the package keeps canonical files in place and settings point at them.

5. Add focused tests around parser normalization, envelope wrapping, auth decisions, team spec parsing, config validation, and reconnect target selection. The bug catalog shows exactly where tests would pay rent.

6. Split `src/cli.mjs` along real product boundaries: messaging/parser dispatch, tmux session lifecycle, team/group loading, backend flag translation, bridge management, and auth/config. Do this as a mechanical extraction after tests, not as a rewrite.

7. Replace or explicitly constrain the YAML parser. Either use a real YAML dependency or document and test the supported subset as "team spec YAML subset." Given the package already depends on MCP SDK and zod, one small YAML dependency is not a philosophical problem.

8. Make logging a product setting, not an incidental file append. Document retention, default path, disable switch, and whether remote messages are logged. Consider redaction only after the trust model is settled.

9. Only after those moves, consider an A2A-facing adapter. Start with an Agent Card that honestly declares a custom binding and a bridge-specific skill. Do not implement fake Tasks unless you are willing to persist task state and expose status honestly.

What I would not spend time on now: full A2A conformance, gRPC, push notification config, signed Agent Cards, multi-modal `Part` support, or artifact stores. Those are A2A product requirements, not terminal-bridge requirements. They become worth doing only if the project explicitly decides to become an A2A service endpoint. Right now, the better product is the bridge: small, local-first, direct, operationally honest, and sharp about the actual way coding agents collaborate today.
